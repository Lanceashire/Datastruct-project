# 图书馆数据模型与哈希索引改进说明

## 一、旧模型的问题

旧版 `Book`同时表示一种书和图书馆里的多本实体书：

```c
Book {
    bookId;
    bookName;
    author;
    category;
    stock;
}
```

这个模型只能表达“某种书还有几本”，不能表达：

- 读者具体借走了哪一本；
- 相同书籍的不同馆藏编号；
- 不同副本所在位置；
- 某一本书损坏或遗失；
- 库存数字与实际副本是否一致。

现实图书馆中的“书目编号”和“馆藏编号”不是同一个概念。

## 二、改进后的两层模型

### 1. Book：书目层

`Book`表示一种确定版本的书：

```c
typedef struct {
    int  catalogId;
    char isbn[20];
    char bookName[50];
    char author[30];
    char publisher[40];
    char category[20];
    int  borrowCount;
} Book;
```

ISBN 是业务唯一标识。相同 ISBN 不能重复建立书目。

### 2. BookCopy：实体馆藏层

`BookCopy`表示图书馆实际拥有的一本书：

```c
typedef struct {
    int  copyId;
    int  catalogId;
    int  status;
    char location[30];
} BookCopy;
```

不同副本拥有不同 `copyId`，但可以指向同一个 `catalogId`。

库存不再作为一个容易失真的数字保存，而是通过统计副本得到：

```text
总馆藏 = 指向该 catalogId 的全部 BookCopy 数量
可借数量 = 状态为 COPY_AVAILABLE 的 BookCopy 数量
```

### 3. Borrow：关联具体馆藏

借阅记录保存 `copyId`，因此可以准确修改某一本实体书的状态：

```text
借阅：COPY_AVAILABLE → COPY_BORROWED
归还：COPY_BORROWED  → COPY_AVAILABLE
```

## 三、顺序表与哈希表协作

系统没有抛弃顺序表。四类业务对象仍保存在连续数组中，因为顺序表适合：

- 按顺序展示全部数据；
- 覆盖写入 TXT 文件；
- 复制后执行冒泡排序；
- 数据量有限的课程实验。

哈希表作为额外索引，解决编号查询和 ISBN 查重需要反复遍历的问题。

## 四、哈希实现

哈希表采用固定容量数组和开放寻址法。

### 1. 哈希函数

字符串键使用 DJB2 思想计算哈希值：

```c
hash = hash * 33 + current_character;
```

整数编号先转换为字符串，因此所有索引共用同一套哈希实现。

### 2. 冲突处理

发生冲突时使用线性探测：

```text
原位置被占用 → 检查下一个槽位 → 直到找到空槽位或相同键
```

### 3. 索引重建

顺序表删除元素后，后续元素下标会发生变化。为了保证哈希表中保存的数组下标正确，系统会重建对应索引。

文件只保存业务数据，不保存哈希表。启动后根据顺序表重新建立索引。

## 五、复杂度变化

| 操作 | 旧版 | 改进版 |
|---|---:|---:|
| 按书目编号查询 | `O(n)` | 平均 `O(1)` |
| 按 ISBN 查询和查重 | 未实现 | 平均 `O(1)` |
| 按馆藏编号查询 | 未实现 | 平均 `O(1)` |
| 按读者编号查询 | `O(n)` | 平均 `O(1)` |
| 按借阅编号归还 | `O(n)` | 平均 `O(1)` |
| 按书名模糊查询 | `O(n)` | `O(n)` |
| 全量展示 | `O(n)` | `O(n)` |
| 借阅排行 | `O(n²)` | `O(n²)` |

书名模糊搜索和全量展示仍需要遍历，这是符合业务特点的。

## 六、与原实验报告的关系

原报告中的顺序表增删改查、冒泡排序、文件持久化思想仍被保留。发生变化的是业务建模：

- `bookId`改为书目编号 `catalogId`；
- 删除手工维护的 `stock`；
- 新增 `BookCopyList`；
- 借阅记录由关联书目改为关联馆藏副本；
- 新增 ISBN 和多张哈希索引。

因此原报告适合作为第一版设计记录，本说明描述当前改进版本。
